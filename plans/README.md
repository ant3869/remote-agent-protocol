## Direction options not scheduled

- The authenticated remote-agent protocol in `docs/architecture.md` remains the strongest next major feature. Start only after the target machine interfaces and authentication method are known; capability discovery and heartbeat should be part of that protocol, not a second ad-hoc launcher.
- An RTVI/web client is intentionally deferred by the architecture until another device must control the app. The current local Tk workflow does not justify that extra surface yet.

## Findings considered and rejected

- Diagnostics include conversation excerpts and task text: intentional, local, and explicitly warned at export; automatic redaction would make support bundles less useful and cannot reliably recognize arbitrary secrets.
- Host-repository before/after snapshots can attribute unrelated edits to a job: this is a warning-only safety signal, so false positives are preferable to silently missing possible agent changes.
- Split the Tk GUI into more classes: not worth doing. The event loop, agent panel, memory panel, config panel, and theme are already separated at useful boundaries.
- Add another storage abstraction for the four small JSON files: rejected as YAGNI; their schemas and failure policies differ, and a shared wrapper would save little code.
